## 1. 사용한 AI 도구

| 도구                                         | 용도                                    | 협업 방식                   |

| **Antigravity (Claude Sonnet 3.7 Thinking)** | 코드 리팩토링, 컴포넌트 분리, 패턴 설계 | 설계 보조 및 코드 리뷰 위주 |

> [!NOTE]

## 2. 인간의 판단에 의한 핵심 수정 사례

### 사례 1: 하이브리드 렌더링 최적화

AI는 모든 페이지와 컴포넌트에 `"use client"`를 기본으로 생성하여 Next.js의 RSC 이점을 무시했습니다.

- **AI 제안**: 전체 CSR 구조로 강제.
- **인간의 수정**: 데이터 페칭은 **서버 컴포넌트(RSC)**에서 처리하고, 실제 인터랙션이 필요한 부분만 클라이언트로 분리하여 JS 번들 크기를 줄이고 SEO/FCP를 확보했습니다.

### 사례 2: 클라이언트 라우터 캐시 동기화

Form 제출 후 서버 데이터는 갱신되나 브라우저 UI가 즉각 업데이트되지 않는 문제를 해결했습니다.

- **문제 상황**: AI 리팩토링 코드가 브라우저의 클라이언트 캐시 갱신을 누락함.
- **의사 결정**: `useRouter().refresh()`를 적절한 시점에 주입하여 서버 상태와 클라이언트 UI 간의 100% 동기화를 보장했습니다.

### 사례 3: 사용자 경험(UX) 디테일 보완

AI가 간과한 실무적인 UX 처리를 직접 보완했습니다.

- **정밀한 입력 제어**: `input type="number"` 대신 `type="text"` + `inputMode="numeric"` 조합을 택해 천 단위 콤마 포맷팅을 지원하면서도 안정적인 데이터 바인딩을 구현했습니다.
- **시각적 피드백**: 활성/비활성 상태에 따른 `cursor` 스타일링을 정교하게 적용하여 사용자 인터랙션의 명확성을 높였습니다.

### 사례 4: 실무 환경 확장 (다크모드 & 일괄 삭제)

요구사항을 넘어 실제 서비스 운영 관점에서 필수적인 기능을 추가했습니다.

- **Advanced Dark Mode**: 단순 배경색 변경이 아닌, **Recharts 내 텍스트와 그리드까지** 테마 상태에 동적으로 반응하도록 설계했습니다.
- **Batch Management**: 캠페인 일괄 상태 변경 및 일괄 삭제 기능을 추가하여 관리자의 업무 효율성을 고려했습니다.

### 사례 5: 매직 넘버 전역 리팩토링

컴포넌트 내부에 산재한 하드코딩된 수치들을 중앙 집중 관리 구조로 변경했습니다.

- **의사 결정**: `shared/constants/` 하위에 도메인별 상수를 정의하여, 한 곳의 수정으로 전체 시스템의 디자인/정책 변경이 가능하도록 견고한 구조를 구축했습니다.

## 3. 기술적 설계 원칙

### 🏗 설계 패턴: SRP & Compound Component

- **Micro Component 분리**: 거대한 컴포넌트를 기능별(DateRange, FilterGroup 등)로 쪼개어 단일 책임 원칙(SRP)을 고수했습니다.
- **Compound Component (`Table`)**: 높은 유연성을 위해 테이블 구조를 서브 컴포넌트 조합 방식으로 구현했습니다.

### 📂 Feature-based Architecture (기능 기반 아키텍처)

기술 계층이 아닌 **도메인 기능(Feature)** 중심으로 폴더를 구성하여 응집도를 높였습니다.

| 계층            | 역할                                                   |
| :-------------- | :----------------------------------------------------- |
| **`app/`**      | 라우팅 및 레이아웃 구성                                |
| **`features/`** | 각 도메인별(Campaign, Dashboard, Filter) 캡슐화된 로직 |
| **`shared/`**   | 전역 공용 컴포넌트, 유틸리티, 타입                     |

## 4. 상태 관리 및 동기화 설계

### 🛠 Global Filter — Zustand SSOT

- `useFilterStore`를 **단일 진실 공급원(SSOT)**으로 삼아 대시보드의 차트와 테이블이 동일한 필터 상태를 실시간으로 공유하고 양방향 연동되도록 설계했습니다.

### 🌐 실시간 데이터 정확성 — `no-store` 전략

- **배경**: 초기 설계에서는 Next.js 16의 `force-cache`와 `updateTag`를 활용한 성능 최적화를 고려했으나, 로컬 개발 환경(`json-server`) 및 과제 검토 환경의 특수성을 고려하여 **항상 최신 데이터를 보장하는 `no-store` 방식**으로 최종 결정했습니다.

- **결과**: Next.js의 캐싱 옵션을 제거하여 가장 신속하고 정확한 데이터 갱신 구조를 확보했습니다.

### ⚡ 차트 렌더링 최적화 — Parallel Routes

- **배경**: 대시보드의 차트 라이브러리(Recharts)는 번들 크기가 크고 초기 렌더링 시 연산량이 많아 전체 페이지 로딩 지연을 초래할 수 있습니다.
- **적용**: Next.js의 **Parallel Routes**를 활용하여 `@charts`와 `@table` 슬롯을 분리하여 레이아웃을 구성했습니다.
- **결과**: 두 영역이 독립적으로 로딩되도록 하여, 데이터가 무거운 차트가 전송/렌더링되는 동안에도 다른 레이아웃이 먼저 사용자에게 노출되는 점진적 렌더링 효과를 통해 체감 성능을 향상시켰습니다.
